# Система аутентификации и авторизации

## 1. Модель данных (User)

Используется стандартная модель `User` Django с дополнительной настройкой.
- `email`: используется в качестве уникального идентификатора для входа (username не используется).
- `password`: хранится в захэшированном виде.
- `is_active`: булево поле (True/False). Используется для мягкого удаления аккаунта. Если False:
    - Пользователь не может войти в систему (аутентификация запрещена).
    - Аккаунт считается удаленным, но данные сохраняются в БД.
- `is_staff`: стандартное поле Django, используется для доступа в админ-панель.
- `is_superuser`: стандартное поле Django.

## 2. Модель данных (Система разграничения доступа)

Архитектура комбинированная: RBAC (Role-Based Access Control) + флаг администратора.

**2.1. Таблица `Role` (Роли)**
Хранит список ролей, доступных в системе.
- `id`: уникальный идентификатор.
- `name`: название роли (например, "Администратор", "Менеджер", "Пользователь").
- `description`: описание роли (необязательное поле).

**2.2. Таблица `Permission` (Разрешения)**
Хранит список всех возможных действий (привилегий) в системе. Привязаны к конкретному ресурсу (модели) или действию.
- `id`: уникальный идентификатор.
- `codename`: уникальный строковый код разрешения (например, `user_create`, `document_view`). Используется для проверки в коде.
- `description`: описание того, что дает это разрешение.

**2.3. Таблица `RolePermission` (Разрешения роли)**
Связующая таблица (Many-to-Many) между `Role` и `Permission`. Определяет, какой набор разрешений соответствует каждой роли.
- `id`: уникальный идентификатор связи.
- `role`: внешний ключ к таблице `Role`.
- `permission`: внешний ключ к таблице `Permission`.

**2.4. Таблица `UserRole` (Роли пользователя)**
Связующая таблица (Many-to-Many) между `User` и `Role`. Определяет, какие роли назначены конкретному пользователю.
- `id`: уникальный идентификатор связи.
- `user`: внешний ключ к модели `User`.
- `role`: внешний ключ к таблице `Role`.

## 3. Логика проверки доступа

При запросе к любому защищенному ресурсу (View) происходит следующее:
1.  Из запроса извлекается пользователь (по сессии или токену).
2.  Если пользователь не аутентифицирован (не передан токен/сессия) -> **ошибка 401**.
3.  Проверяется флаг `is_active` у пользователя. Если `False` (аккаунт мягко удален) -> **ошибка 401** (как при неаутентифицированном доступе).
4.  Если пользователь аутентифицирован и активен:
    *   Проверяется, является ли он суперпользователем (`is_superuser=True`). Если да -> доступ **РАЗРЕШЕН** (полный доступ ко всем ресурсам без проверки прав).
    *   Если не суперпользователь, то система собирает все разрешения (`Permission.codename`) из всех ролей (`UserRole` -> `Role` -> `RolePermission`), назначенных пользователю.
5.  Проверяется, содержит ли список разрешений пользователя то разрешение, которое требуется для доступа к текущему ресурсу (например, `user_update`).
6.  Если разрешение найдено -> доступ **РАЗРЕШЕН**.
7.  Если разрешение не найдено -> **ошибка 403**.
